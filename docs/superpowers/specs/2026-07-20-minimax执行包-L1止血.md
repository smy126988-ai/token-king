## 任务 T1:allCases 防线(半天,先做,最安全)

### 背景
`ConfigInfoProviderTests.swift:155 testAllListedProvidersReturnExplicitEntries` 本意是「检查每个 provider 都有明确配置、不走 default 兜底」。但它用的是**硬编码数组**(`:159 let identifiers: [ProviderIdentifier] = [...]`,手列 27 个),不是 `ProviderIdentifier.allCases`(实际 31 个)。后果:新加 provider 忘了同步这个数组,测试也不报警——防线有洞。

### 改什么
1. 打开 `CopilotMonitorTests/ConfigInfoProviderTests.swift`,定位 `:159` 的 `let identifiers: [ProviderIdentifier] = [ ... ]`
2. 把这个硬编码数组替换为 `let identifiers = ProviderIdentifier.allCases`
3. 如果 `ProviderIdentifier` 还没有 `allCases`(没 conform `CaseIterable`):去 `CopilotMonitor/Models/ProviderProtocol.swift` 给 enum 加 `CaseIterable` conformance

### 验收(必须全绿)
```
cd CopilotMonitor && xcodebuild test -scheme CopilotMonitor -destination 'platform=macOS' 2>&1 | tail -40
```
- **预期**:该测试可能先**转红**(因为 31 个里有 4 个之前漏在硬编码数组外、真的走了 default)。若转红,说明防线抓到了真实缺口——把这些 provider 在 `configInfo` 里补上明确分支,再跑到绿。
- 若一开始就绿,说明 27 个之外的 4 个恰好有兜底,仍需确认它们返回的不是通用占位文案。

### 提交
```
git add CopilotMonitorTests/ConfigInfoProviderTests.swift CopilotMonitor/Models/ProviderProtocol.swift
git commit -m "test: allCases 驱动 provider 配置守卫测试,堵漏同步缺口"
```

## 任务 T3:删除 7 个死方法(半天)

### 背景(已核实:各只 1 处出现 = 定义处,零调用)
以下 7 个方法在 `CopilotMonitor/CopilotMonitor/App/StatusBarController.swift` 里,全库 grep 各只出现 1 次(即定义处,无任何调用),是 Copilot 单 provider 时代遗留:
`clearCaches` `saveHistoryCache` `loadDemoData` `updateUIForSuccess` `handleFetchError` `signInClicked` `openBillingClicked`

### 改什么(逐个,不要批量)
对每个方法:
1. 先**再次核实零调用**(防止我给的清单过期):
   ```
   cd CopilotMonitor && grep -rn "\bclearCaches\b" CopilotMonitor --include="*.swift"
   ```
   把 `clearCaches` 换成对应方法名。**只应出现 1 行(定义处)**。若出现 >1 行,说明有调用,**停下报告,不要删**。
2. 确认零调用后,删除该方法的完整定义(从 `func xxx` 到对应闭合 `}`)。
3. 因为是删代码不是删文件,直接在 StatusBarController.swift 内删除方法体即可(无需 mv 整文件)。
