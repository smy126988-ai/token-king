# Token King 产品体验与文档设计

## 1. 目标

让用户在打开菜单后 10 秒内理解四件事：本月用了多少 Token、实际按量支出多少、如果按 API 计价值多少钱、订阅每月花多少钱，并知道每个数字是否完整和新鲜。

## 2. 信息架构

保留不可变组标题：

- `按量付费：¥XX.XX`
- `额度状态：¥XXX/月`

顶层动态摘要按以下顺序：

1. `本月 Token：10.5B`
2. `全局统计` 子菜单
3. 现有按量付费组
4. `本月 API 折算：¥XX,XXX（已计价 92%）`
5. 现有额度状态组

实际支出、API 折算和订阅成本使用明确不同的标题与帮助说明，不合并成一个“总成本”。

## 3. 统计五态

每个统计区统一支持：

| 状态 | 文案要求 |
|---|---|
| 加载中 | 说明正在读取哪些本地来源 |
| 无历史 | 说明尚无记录以及何时会产生 |
| 部分数据 | 显示 coverage、缺失来源和最后成功时间 |
| 失败 | 原因、下一步、缓存是否可用 |
| 完整 | 数值、更新时间、估算假设入口 |

缓存结果必须显示最后成功时间，不能与新鲜结果无差别。

## 4. 明细与格式

- Token formatter 增加 B/T 级别，避免 `1653.6M`。
- Provider 明细展示 input、cache read、cache write、output、reasoning 和按模型成本。
- 未计价模型仍显示 token；用系统图标/辅助文字表达，不仅靠星号 tooltip。
- F1/F3/F4 用户文案统一中文；模型、API、Token、Provider 品牌名保留英文。
- 所有自定义行使用 `MenuDesignToken` 和右对齐布局，不用空格拼表格。

## 5. Provider 显隐

启用现有状态能力并增加 `设置 → 显示的服务商`：

- 勾选状态持久化；
- 主菜单、刷新任务和状态栏候选同时生效；
- 至少保留一个 Provider，或展示明确空状态；
- “尚未配置”保留折叠入口，但不让大量无关 Provider 抢占主菜单。

## 6. README、截图与 backlog

- README 第一屏说明 Provider 主视角、跨工具 Token/额度/成本和本地数据隐私。
- 所有安装、clone、Issue、Release、badge 链接指向 Token King fork；上游只留 Credits。
- 使用 demo/testing mode 生成当前中文截图，覆盖四个核心数字和设置入口。
- `docs/需求池.md` 只做 inbox；正式状态统一在 backlog，ID 不重号。
- 已完成项附 commit/test，未完成项附验收标准。
- README 的 Provider 列表和版本由脚本校验，防止再次漂移。

## 7. 验收

- demo fixture 截图能在 10 秒内回答四个核心问题。
- 主菜单、F1/F3/F4 和错误态通过中文字符串扫描。
- 五态都有单元或 UI 测试。
- Provider 显隐在重启后保持，且不会继续刷新已禁用 Provider。
- README 从零开始可在五分钟内完成安装并理解核心口径。
