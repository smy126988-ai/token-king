## 5. Schema migration (v1 → v2)

`TokenUsageStore.sqls` now declares 7 statements executed in order on every `init`:

```
1. CREATE TABLE IF NOT EXISTS token_events     (unchanged)
2. CREATE TABLE IF NOT EXISTS month_aggregates (unchanged)
3. CREATE TABLE IF NOT EXISTS day_aggregates   (NEW)
   (provider, model, day, input, output, cache_read, cache_write, reasoning, last_updated,
    PRIMARY KEY (provider, model, day))
4. CREATE TABLE IF NOT EXISTS model_pricing_cache (unchanged)
5. CREATE TABLE IF NOT EXISTS schema_version   (unchanged)
6. INSERT OR IGNORE INTO schema_version VALUES (1)   (existing row)
7. INSERT OR IGNORE INTO schema_version VALUES (2)   (NEW)
8-9. CREATE INDEX IF NOT EXISTS idx_token_events_* (unchanged)
10.  CREATE INDEX IF NOT EXISTS idx_day_aggregates_day (NEW)
```

**Upgrade safety:**
- Existing v1 DBs add `day_aggregates` (empty) and `schema_version` row 2 transparently — `CREATE TABLE IF NOT EXISTS` + `INSERT OR IGNORE` are no-ops on existing rows.
- No data migration required; `day_aggregates` starts empty and is populated by `RefreshActor.tick()` → `refreshDayAggregates()` for today.
- Old v1 DBs do not have historical day data; F1 daily list shows current month only (see §6).

## 6. Known limitations

1. **Kimi CLI / Kimi Code CN/Global split is partial.**
   - `OpenCodeExtractor` distinguishes CN vs Global via the upstream `providerID` field in JSON.
   - `KimiCLILegacyExtractor` and `KimiCodeExtractor` hardcode `providerID: "moonshot"` — there is no CN/海外 signal in their raw data without env-var sniffing.
   - Net: events from Kimi CLI / Kimi Code go to `.kimi` (Global) bucket regardless of actual deployment region. Acceptable for v1.

2. **F1 daily list scope is current month only.**
   - `fetchDayAggregates(yearMonth:)` defaults to `store.currentYearMonth() = YYYY-MM` in UTC.
   - No 12-month rolling window in v1; "last 30 days" is a sensible v2 add.

3. **F3 5h 桶明细 v1 only shows current state + reset time.**
   - No historical snapshot storage. The 5h percent shown is the latest fetch value.
   - "5h 桶 chart over time" is v2.

4. **F4 quota row hardcodes "/月" Chinese suffix** in `displayText` (intentional: this is the Token King fork and the user is fine with bilingual strings).

5. **`Helpers/TimeZone+UTC.swift` is untracked** — pre-existing infra written in an earlier session. This blocks a clean checkout. See §9.

6. **Pre-existing dirty working tree:**
   - 28+ uncommitted files predate this F1/F3/F4 work (e.g. `TokenExtractor/KimiCLILegacyExtractor.swift`, `Services/UsagePredictor.swift`, several provider files).
   - Some uncommitted edits were touched as scope creep during this work: `AppDelegate.swift` (B39 multi-display status item bridge), `TokenUsageStore.initError` surfacing in `refreshMonthlyTotalsCache` / `refreshTokenStatsCache` (B54).
   - These clean commits should be landed separately — see §9.

## 10. Quick reference for next session

- **All new helper files are pure functions** — `TokenStatsAggregator.snapshot(...)`, `TokenUsageFormatter.format(...)`, `WindowConstants.TimeWindow.*`. No actor / SQLite setup needed to test them.
- **`StatusBarController.createGlobalStatsSubmenu(snapshot:)` is `static`** — pass a hand-built `TokenStatsAggregator.Snapshot` from tests, no async I/O in tests.
- **`StatusBarController.refreshTokenStatsCache()`** is the entry point; `refreshMonthlyTotalsCache()` runs first. Both check `await store.initError != nil` early.
- **`day_aggregates` key is `(provider, model, day)`** — same prefix pattern as `month_aggregates (provider, model, year_month)`. Adding a new aggregation period = add a new table with same shape + a new prefix format.
- **F2b Provider enum keys**: `kimi`, `kimiCN`, `claude`, `codex`, `zai`, `nanoGpt`. Anything else maps to `.nanoGpt` fallback (logged warning).
