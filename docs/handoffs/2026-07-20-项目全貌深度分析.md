### 为什么修改危险

### ⚠️ 拆分红线(不可碰)
`pendingStatusItem` 队列、`attachTo(_:)` 桥接、`renderStatusItemImage()` **不是 hack,是 macOS 26.x 菜单栏能显示的唯一路径**(详见 `docs/handoffs/2026-07-04-statusbar-architecture-problem.md`)。拆类时必须原样保留这条桥接链。

3. **菜单构建**(`updateMultiProviderMenu` + `create*` 系列)→ `MenuAssembler`
4. **死缓存方法**直接删

## 2. Provider 横向扩展面 —— 具体是什么问题

`ProviderIdentifier` enum(`Models/ProviderProtocol.swift:45`)**31 个 case**。文档 `docs/adding-a-provider.md` 声称加一个改 7 处,**实测低估**:对该 enum 的穷举 switch 跨 **8+ 文件、约 20+ 处**。

### 新增一个 provider 的真实改动清单
1. 新建 `Providers/FooProvider.swift`(实现 `ProviderProtocol`)
2. `ProviderProtocol.swift:45` enum 加 case + 同文件 6 个穷举 switch(`family :78`、`region :117`、`displayName`、`shortDisplayName`、`iconName`、`isEnabled`)
3. `Services/TokenManager.swift`:auth 字段 + `getFooAPIKey()`
4. `Services/ProviderManager.swift` `makeDefaultProviders()`:注册实例
5. `Models/SubscriptionSettings.swift:340` 及 11 个 switch
6. `xcodeproj/project.pbxproj`:6 条手动注册(**最易漏**)
7. Tests

### bug 温床:两类 switch
- **无 default(编译器强制穷举)**:`usagePercentCandidates :1397`、`iconForProvider :3805`、enum 内 `displayName/family`。忘改 = 编译失败,**安全**。
- **有 default(静默兜底)= 温床**:`f2bProviderRaw :1296 default:return nil`、`configInfo :3532 default:return 通用占位`、`region :117 default:.global`、`StatusBarRefreshAndF2b :477/:498`。忘改 → **编译通过 → 运行时走 default → 静默错误**。这正是 B02/B16-B29 那批 wiring bug 的机制。

### switch 分布(grep 实测)
- `StatusBarController.swift`:6 处(`f2bProviderRaw :1287/:1306`、`usagePercentCandidates :1397`、`multiAccountBaseName :3045`、`configInfo :3473`、`iconForProvider :3805`)
- `StatusBarUsageDisplay.swift :126`(注:此文件整体 #if false 死代码)、`StatusBarRefreshAndF2b.swift :468/:488`
- `ProviderMenuBuilder.swift :69`、`SubscriptionSettings.swift`(多处)、`ProviderUsage.swift`、`ProviderResult.swift`

### 兜底测试有缺口
`ConfigInfoProviderTests.swift:155 testAllListedProvidersReturnExplicitEntries` 用的是**硬编码 27 个 identifier 数组**,不是 `ProviderIdentifier.allCases`——新加 case 忘同步数组,测试也不失败。`ProviderEnumAlignmentTests.swift:193` 用 allCases 较可靠;`f2bProviderRaw` 的 default 无 allCases 兜底。

### 收敛方案
把 provider-specific 数据下沉进 `ProviderProtocol`(各 Provider 自带 `iconName/configInfo/f2bRaw/usageWindows/subscriptionPresets/menuRows`),外部 switch 改为属性读取。**立即可做的低成本止血**:所有 `testAll...` 守卫测试从硬编码数组改为 `allCases` 遍历,马上堵住"漏同步"缺口。

## 3. 数据正确性 —— 深度分析(含前提推翻)

### 少算 ~84% 的真实根因
**不是抽取层 bug,是聚合读取路径 bug。** UI"本月 Token"原读 `day_aggregates` 表,该表**增量维护**——每 tick 只重写当天,历史天从不回填。全新 DB / schema reset / app 前几天没运行时,月度求和严重偏小。
- 证据:`TokenUsageStore.swift:302-310` 注释明写 "day_aggregates is INCREMENTAL ... can underreport a full month's total"
- 实测(`docs/handoffs/2026-07-13-p0-batch1-final.md:15`):修复后 UI 从 **2933M(24.9%)→ 11777M(100%)**,少算约 75%("84%"是 issue 标签,量级随残留历史天数浮动)
- 修法:改读 `month_aggregates`(每 tick 由 token_events 全量重导),经 `fetchMonthAggregatesSum()`(`TokenUsageStore.swift:311`)取值,以 `monthTotalOverride` 传入 `TokenStatsAggregator.snapshot()`(`:40`)。调用点 `StatusBarRefreshAndF2b.swift:425-432`、`StatusBarController.swift:1246-1253`

### 关键反转:cacheRead delta 假设已被推翻
早前 7-10 审计假设"cacheRead 是 session 累计需算 delta",**后被推翻**:
- **OpenCode**(`OpenCodeExtractor.swift:37-41`):曾做累计转 delta,反把 MiniMax-M3 从 ~18 亿砍到 ~80M(**1/20**),已 revert,现按 per-request 原值求和
- **Codex**(`CodexExtractor.swift:30`):不做 delta,改用**去重**(与紧邻上一条 5 字段全等则 skip,对应 ccusage PR#824,避免 CLI 重发快照致 cache_read 虚高 ~30%);仅老版 Codex(无 `last_token_usage`)才走 `total_token_usage` 累计 delta 兜底(`:204`)
- **KimiCode**(`KimiCodeExtractor.swift:62`):直读 `inputCacheRead`,无 delta

两个真正少算源:①UI 读 day_aggregates(已修) ②曾过度 delta 把 OpenCode 砍到 1/20(已 revert)

### 修复状态 —— 重要
| 项 | 状态 |
|----|------|
| 月度少算修复 `464f6e5`(month_aggregates override) | ✅ **已在 main**(HEAD b8bef40 祖先) |
| 回归测试 | ✅ `TokenUsageStoreMonthAggregatesSumTests`、`TokenStatsAggregatorTests`;整体 695 pass / 19 skip / 0 fail |
| **codex/token-king-95 分支** | ❌ **不含此修复**(从更早 e8a2f31 分叉,15 commit 全是测试隔离,不碰数据管线)→ **出包会回归** |

### 仍未解决的数据正确性隐患
1. **Codex 去重只比紧邻上一条**(`CodexExtractor.swift:150-163`)——非相邻重复、事件乱序会漏检,无排序保证
2. **day_aggregates 仍只重写当天**——today/week UI 路径仍读它,跨越未运行日的"本周"仍可能少算(month 靠 override 绕过,day/week 没绕)
3. schema 缺 `max_input_in_event`——MiniMax-M3 512K 长上下文档位无法按行计价($290 名义 vs $581 最坏)
4. KimiCLI legacy:token 全塞 output,无 input/cache 拆分,靠 file mtime
5. **95 分支缺 month 修复**——出包路径风险
6. NanoGPT/ZAI 快照路径无 7 月真实数据,未经真实验证
